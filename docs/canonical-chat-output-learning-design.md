# GAG / GAE Canonical Chat Output & Learning Core

Status: implementation contract
Target score: 95/100 or higher
Scope: every ChatGPT conversation in which the GAG or GAE connector is available

## 1. Purpose

The core must make GAG and GAE behave consistently across conversations, expose useful progress without relying on MCP cards, produce one deterministic final structure, and improve from the relationship between each assistant output and the user's next message.

This is not a generic memory system. It is a bounded output-optimization loop:

1. capture the current request;
2. pair it with the previous completed output in the same conversation;
3. classify the next-message reaction;
4. update stable output guidance and predicted next intents;
5. start the task with the learned guidance;
6. complete only through the canonical finalizer.

## 2. Hard constraints

- MCP cannot passively read all ordinary ChatGPT messages. Full automatic capture therefore requires either GEX DOM capture or a mandatory model-to-tool handoff.
- The core must never claim that learning happened when neither path supplied the message pair.
- Raw conversations must remain local/private by default. Shared synchronization stores bounded summaries and classifications, not secrets or complete transcripts.
- GAG and GAE are separate runtimes. Cross-runtime learning requires an explicit shared snapshot mechanism.
- A final format cannot be guaranteed by prompt text alone. A server-generated finalizer is the source of truth.

## 3. Architecture

### 3.1 Conversation turn ingestion

Primary path: `begin_task`

- Called once before substantive work in every GAG/GAE task.
- Receives the current user request and optional task category.
- Reads the previous pending output for the same conversation.
- Uses host conversation metadata first. When that metadata or the MCP transport changes, an opaque `continuityKey` returned by the previous finalizer preserves the same-Chat chain.
- If one exists, stores the pair `previous output summary -> current user request`.
- Marks the pending output as consumed with a bounded tombstone so another runtime cannot learn the same pair twice.
- Classifies the reaction and runs analysis when due.
- Starts progress and returns task-category-specific guidance plus predicted next intents.

Secondary path: GEX learning sync

- GEX may submit exact prompt/assistant/follow-up loops.
- The server normalizes conversation keys and stores them in the same learning store.
- Deterministic IDs remove identical repeats; same-conversation/same-follow-up proximity checks remove duplicates observed once by GEX and once by `begin_task`.

### 3.2 Task state machine

Allowed states:

`idle -> running -> paused -> running -> completed|failed`

Rules:

- `begin_task` creates or resumes one task record per conversation.
- `report_progress` updates only meaningful milestones, blockers, or forecast changes.
- `finalize_task` is the only successful completion path.
- Opening a workspace may create a fallback running record, but `begin_task` adopts it rather than creating a duplicate.
- A stale running record becomes paused, not completed.

### 3.3 Canonical visible output

During work, visible ordinary chat text must contain:

- task label;
- current progress;
- current action;
- completed evidence;
- next action;
- blocker/risk;
- elapsed time and ETA when available.

The final response is generated by the server with exactly these sections:

1. `## 完了結果`
2. `## 変更`
3. `## 検証`
4. `## 残り`
5. `## 次に起こりそうなこと`
6. `## 実行情報`

`次に起こりそうなこと` contains up to three learned predictions. Empty sections use `なし`.

### 3.4 Card policy

Cards are evidence surfaces, not the main narrative.

- Default state is collapsed.
- One card is justified only when its result may change the next action, isolate a failure, request approval, or expose a reviewable diff.
- Progress tools use compact state cards and do not append a duplicate usage table.
- Independent inspection and verification steps are batched.
- The finalizer card contains only the completion state; the canonical response appears in normal chat.

### 3.5 Learning record

A bounded record stores:

- runtime and conversation key;
- task category;
- previous output summary and section signature;
- predicted next intents made at completion;
- actual next user input summary;
- classified reaction;
- matched prediction rank;
- detected improvement signals;
- timestamps and source (`begin_task` or `gex`).

Raw text is bounded and secret-like values are redacted before persistence.

### 3.6 Reaction classes

The deterministic classifier uses these classes:

- `approval`: explicit completion/thanks/acceptance;
- `continuation`: natural next step without correction;
- `refinement`: adds constraints or asks for a better version;
- `correction`: says the result is wrong, unchanged, missing, or misunderstood;
- `verification`: asks whether it is complete, correct, deployed, or tested;
- `new_topic`: low lexical relationship to the previous output;
- `unknown`: insufficient evidence.

The classifier exposes confidence and matched evidence. It must not present inference as fact.

### 3.7 Periodic analysis

Analysis runs when either condition is met:

- 10 new complete loops have arrived; or
- 6 hours have elapsed since the last analysis and at least one new loop exists.

It calculates:

- reaction rates by task category and output signature;
- correction/refinement hotspots;
- prediction top-1 and top-3 accuracy;
- recurring next intents;
- stable guidance only when supported by at least three samples;
- regressions compared with the previous analysis window.

The resulting profile is bounded to actionable rules. A single reaction never becomes a permanent rule.

### 3.8 Cross-runtime synchronization

Each runtime writes a local atomic JSON document with mode `0600`.

A shared snapshot contains only normalized, bounded learning records, consumed-output tombstones, and aggregate guidance. It uses one file per runtime so concurrent writes cannot overwrite another runtime. Pull merges by deterministic record ID and applies tombstones before exposing pending outputs; publish never deletes remote records.

Synchronization is local-first. Progress-history and output-learning pull/publish operations run asynchronously and update their status after the user-facing response. Each remote operation is cached and bounded by an 8-second default timeout. A slow remote therefore cannot block task start or final output. Synchronization failure is non-fatal and must be visible in diagnostics. Local capture continues.

## 4. Quality scoring

Implementation quality is computed from evidence, not self-declared confidence.

| Dimension | Weight |
|---|---:|
| Deterministic final structure and state machine | 20 |
| Turn pairing and reaction classification | 20 |
| Prediction and periodic analysis | 15 |
| Cross-chat and cross-runtime persistence | 15 |
| Card economy and duplicate-output prevention | 10 |
| Privacy, bounded storage, and failure behavior | 10 |
| Automated tests and runtime verification | 10 |

Completion threshold: `>=95` with no critical failure.

Critical failures:

- finalizer can emit an invalid section order;
- current request is paired with the wrong conversation;
- secret-like text is persisted unredacted in a tested case;
- GAG and GAE expose incompatible required tool schemas;
- a failed verification is reported as successful completion.

## 5. Test matrix

### Unit

- reaction classifier for all classes;
- redaction and text bounds;
- deterministic IDs, cross-source deduplication, and consumed-output tombstones;
- pending-output pairing by conversation and `continuityKey`;
- prediction ranking and accuracy;
- periodic analysis thresholds;
- final section order and empty-section handling;
- quality score and critical-failure blocking.

### Integration

- `begin_task -> report_progress -> finalize_task -> begin_task` loop;
- two conversations sharing one transport session remain isolated;
- GEX loop and tool loop deduplicate;
- usage tables appear only in progress/final execution output;
- progress-history pull/publish return without waiting for remote I/O;
- stale task resume;
- shared snapshot merge from GAG and GAE.

### Runtime

- Mac typecheck, focused tests, full tests, build, MCP tool listing;
- EC2 same commit, same focused tests, build, runtime smoke;
- separate ChatGPT conversation invokes required tools and returns the canonical final structure.

## 6. Definition of done

The work is complete only when:

- the weighted evidence score is at least 95;
- required schemas match on GAG and GAE;
- all focused and full tests pass on both runtimes;
- the separate-chat experiment passes;
- its exact ChatGPT link and observed result are reported;
- no production deploy, main push, or destructive operation is performed without explicit authorization.
